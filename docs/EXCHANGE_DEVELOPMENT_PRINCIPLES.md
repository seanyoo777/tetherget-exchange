# 거래소 전용 개발 원칙

이 문서는 **중앙화 거래소(CEX)** 계열 기능을 설계·구현·운영할 때 준수할 원칙입니다.  
공통 생태계 원칙(`docs/COMMON_DEVELOPMENT_PRINCIPLES.md`)과 함께 적용합니다.

---

## 1. 주문·체결 데이터 무결성

주문/체결 데이터는 **절대 꼬이면 안 된다.**  
- 단일 진실 소스는 서버(주문 엔진·원장) 기준.  
- 상태 전이는 명시적 규칙으로만 허용하고, 경쟁 조건·중복 처리를 방지한다.

## 2. 주문 상태값 세분화

아래 상태 모델을 기준으로 설계하고, 프론트·API·DB 필드명을 일치시킨다.

| 개념 | 설명 |
|------|------|
| 접수 | 주문이 유효하게 수락됨 |
| 부분체결 | 일부만 체결됨 |
| 전체체결 | 전량 체결 완료 |
| 취소대기 | 취소 요청 처리 중 |
| 취소완료 | 취소 확정 |

## 3. 오더북 실시간 반영 구조 최적화

- 실시간 깊이·스프레드 반영 시 **지연·역주행·유실**을 최소화한다.  
- 소스(WebSocket/REST 등)와 재연결·스냅샷 정합성 전략을 문서화한다.

## 4. 차트·호가·체결내역 동기화

동일 심볼·동일 시간축 기준으로 **차트 / 호가 / 체결내역**이 논리적으로 어긋나지 않도록 한다.  
- 타임스탬프·심볼·세션 정책을 통일한다.

## 5. 서버 재시작 시 주문 데이터 복구

재시작 이후에도 **미체결·부분체결·원장 상태**를 일관되게 복구할 수 있어야 한다.  
- 영속화·체크포인트·재처리(replay) 가능성을 설계에 포함한다.

## 6. 지갑 잔액 계산 검증

지갑·마진·출금 가능액 등은 **검증 로직·대사(reconciliation)** 로 주기적 또는 이벤트 기준 확인 가능해야 한다.

## 7. 거래 수수료 계산 로그 저장

수수료 산출 근거(메이커/테이커, 등급, 프로모션 등)가 **감사 가능**하도록 로그에 남긴다.

## 8. 관리자 강제 거래 정지

플랫폼 또는 특정 범위에 대한 **강제 거래 정지**를 관리자 도구로 수행할 수 있어야 한다.

## 9. 특정 코인 거래 OFF

심볼(또는 마켓) 단위로 **거래 가능 여부**를 끌 수 있어야 한다.  
- 신규 주문 차단, 기존 미체결 처리 정책을 명시한다.

## 10. 비정상 거래 탐지

이상 패턴(급격한 자금 이동, 반복 취소, 시세 이탈 주문 등)에 대한 **탐지·알림·차단 연동**을 고려한다.

---

## 변경 시 절차

- 기존 기능 삭제 없이 확장하는 것을 원칙으로 한다.  
- 변경 후 **테스트 리포트**를 남긴다(공통 원칙 10항과 동일 취지).
