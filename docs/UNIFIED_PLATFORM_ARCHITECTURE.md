# 통합 플랫폼 아키텍처 (통합 전제 설계)

이 문서는 **개별 제품을 독립 구획으로 고정하지 않고**, 향후 **하나의 통합 플랫폼**으로 합칠 수 있도록 현재 저장소가 따라야 할 **경계·식별자·API·데이터·진화 경로**를 정의한다.  
기존 원칙은 `docs/COMMON_DEVELOPMENT_PRINCIPLES.md`와 동일 선상에서 보완한다.

---

## 1. 목표 상태 (North Star)

- **단일 사용자 계정**: 한 `user_id`로 여러 제품(거래소, P2P, 카드, DAO 등) 접근.
- **공통 운영·감사**: 로그인·거래·관리자 행동 등 **동일한 스키마**로 수집 가능.
- **제품은 조립 가능**: 새 서비스를 **플러그인**처럼 추가하고, 기존 코어는 재사용.
- **점진적 분리**: 지금은 단일 리포지토리라도 **경계를 코드와 타입으로 명시**해, 나중에 패키지/서비스 분리 비용을 줄인다.

---

## 2. 도메인 경계 (Bounded Context)

통합 시 분리·재사용 단위를 다음처럼 가정한다. 이름은 구현체와 1:1일 필요 없고, **책임**이 중요하다.

| 컨텍스트 | 책임 | 이 저장소와의 관계 |
|----------|------|---------------------|
| **Identity & Session** | 로그인, 세션/JWT, 기기, `user_id` 발급·조회 | 현재는 단순화 가능하나, **외부 키는 항상 `user_id`** 로 고정 |
| **Wallet / Ledger** | 잔고·입출금·원장 이벤트 | 거래소 엔진과 분리할 때 **이중 기록 방지**를 위해 출금/잠금은 단일 진입점으로 설계 |
| **Trading / Matching** | 주문, 체결, 심볼, 세션별 매칭 | `backend` 시뮬·매칭 로직은 **“Trading 엔진 어댑터”** 로 간주 |
| **Risk & Compliance** | 한도, 블랙리스트, 거래 정지 | 관리자 휴장·심볼 OFF 등은 **정책 입력**으로 모델링 |
| **Admin & Ops** | 권한 역할, 대시보드, 수동 개입 | 역할은 `COMMON_DEVELOPMENT_PRINCIPLES` 의 관리자 계층과 정렬 |
| **Market Data** | 호가, 지수, 외부 시세 어댑터 | Bitget 등 외부 연동은 **어댑터 인터페이스** 뒤에 둔다 |
| **Notifications** | 이메일, 푸시, 인앱 | 이벤트 구독형으로 붙일 수 있게 **도메인 이벤트** 우선 |

**원칙**: UI·REST 경로만으로는 부족하고, **타입·이벤트 이름·persisted 스키마 버전**에서 경계를 드러낸다.

---

## 3. 식별자와 상관관계 (Integration-ready IDs)

- **`user_id`**: 플랫폼 전역 불변. 다른 시스템의 내부 PK를 노출하지 않고 매핑 테이블로 연결.
- **`request_id` / `correlation_id`**: 관리자·주문·출금 API에 **선택적으로 헤더**로 받아 로그에 전파 (향후 분산 추적과 호환).
- **주문·출금 ID**: 제품별 접두나 네임스페이스를 두면 통합 DB에서 충돌을 피하기 쉽다 (예: `ord_`, `wd_` 또는 ULID).

---

## 4. API 계약 (Contract-first)

- **버전**: 경로 또는 헤더로 `Accept-Version` / `/v1/` 등 **명시적 버전**. 깨지는 변경은 마이너 버전 이상.
- **에러 봉투**: 코드·메시지·필드명을 **공통 스키마**로 정의할 계획 (예: `{ "error": { "code", "message", "details" } }`). 새 엔드포인트는 이 형태로 추가.
- **멱등성**: 입금 확인·출금·주문 생성 등에는 **`Idempotency-Key`** 헤더를 채택할 여지를 남긴다.

런타임 검증 스키마(Zod)는 **`@tetherget/contracts/schemas`** 서브패스에 두어 API 게이트웨이·백엔드만 의존하고, 브라우저 번들에는 포함하지 않는다. 타입·상수는 메인 `@tetherget/contracts`에서 공유한다.

---

## 5. 데이터와 저장소

- **단일 진실**: 금액·수수료·체결은 서버 (`COMMON_DEVELOPMENT_PRINCIPLES` §5).
- **상태 파일·DB**: `exchange-state.json` 같은 로컬 파일은 **개발/단일 노드** 전제. 통합 시에는 **Ledger + Trading** 영속 계층으로 이전하되, **스키마 버전 필드**는 유지해 마이그레이션 경로를 남긴다.
- **감사 로그**: 구조화 로그는 **이벤트 타입** 열거로 확장 가능하게 (로그인, 거래, 관리자 행동 등).

---

## 6. 비동기 통합 (선택적이지만 권장)

제품 간 결합도를 낮추려면:

- **도메인 이벤트** (예: `OrderFilled`, `WithdrawalRequested`)를 내부 버스나 큐로 발행할 수 있는 형태로 모델링.
- 동기 REST만으로도 동작 가능하나, 통합 플랫폼에서는 **이벤트 소싱/감사**와 맞물린다.

---

## 7. 저장소 진화 경로 (이 리포지토리)

단계적으로 다음을 권장한다. 한 번에 전부 할 필요는 없다.

1. **계약 묶음**: 공통 타입·에러 코드는 **`@tetherget/contracts`** (`shared/contracts`) 에서 일원화하고, 프론트(Vite 별칭)·백엔드가 동일 패키지를 참조한다.
2. **영속성 어댑터**: 매칭 엔진·스냅샷 저장을 **인터페이스 뒤**로 두어, 나중에 DB/다중 인스턴스로 교체.
3. **인증 주입**: 세션 검증을 미들웨어 한 곳으로 모아, 통합 SSO 연동 시 교체 용이.
4. **배포 단위**: 최종적으로 `apps/web`, `apps/api-exchange`, `packages/*` 같은 구조로 분리 가능.

`shared/README.md` 에 향후 패키지 목적을 짧게 적어 두었다.

---

## 8. 이 문서의 사용법

- 새 기능 추가 시: **어느 컨텍스트에 속하는지**, **식별자·로그·API 봉투**가 통합 규칙과 맞는지 확인한다.
- 리팩터링 시: “통합 시 한 번 더 옮겨야 할 코드”를 줄이는 방향을 우선한다.
